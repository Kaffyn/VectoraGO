REFACTORING EXAMPLE PROMPTS
=============================

Copy these prompts directly into Claude Code.
Follow the multi-step approach for best results.

---

STEP 1: ANALYZE CURRENT STATE
===============================

@vectora Analyze [ASPECT] in this codebase.

Find and show:
1. How [ASPECT] is currently implemented
2. Inconsistencies (different approaches in different files)
3. What the best practice should be
4. Which files need refactoring
5. Risk level (how critical is this?)

Format as a prioritized list.

---

STEP 2: DESIGN TARGET PATTERN
==============================

@vectora Design a standard [ASPECT] pattern for this project.

Requirements for the new pattern:
1. [REQUIREMENT 1]
2. [REQUIREMENT 2]
3. [REQUIREMENT 3]

Create:
1. Definition of the target pattern
2. Complete working example
3. How it differs from current state
4. Benefits of the new pattern
5. Migration strategy (how to update existing code)

Show complete code snippets.

---

STEP 3: REFACTOR HIGHEST PRIORITY FILES
=========================================

@vectora Refactor [FILE_NAME] to use the [ASPECT] pattern.

The target pattern is defined in Step 2 output.

Show:
1. Current code from [FILE_NAME]
2. Refactored code
3. Changes explained line-by-line
4. How to test these changes
5. Any edge cases to consider

Make it a diff format (old → new).

---

STEP 4: CREATE REFACTORING CHECKLIST
======================================

@vectora Create a refactoring checklist for standardizing [ASPECT].

Based on Step 1, there are [N] files that need updating.

Provide:
1. Complete list of files to refactor
2. For each file:
   - Current pattern it uses
   - Target pattern it should use
   - Complexity (easy/medium/hard)
3. Suggested refactoring order
4. Test strategy for verification

TIPS FOR SUCCESSFUL REFACTORING
================================

✅ DO:
1. Start with Step 1 analysis (understand what you have)
2. Do Step 2 design before any code changes (get pattern right)
3. Refactor one file at a time (Step 3)
4. Use Step 4 checklist to ensure nothing is missed
5. Test after each file (don't batch refactoring)
6. Commit frequently (keep changes small and reviewable)

❌ DON'T:
1. Skip analysis (you might refactor the wrong thing)
2. Change multiple aspects at once
3. Refactor without a clear target pattern
4. Refactor without tests
5. Make refactoring and feature changes together

---

TESTING AFTER REFACTORING
===========================

After refactoring, always test:
1. Unit tests still pass
2. Integration tests still pass
3. Manual smoke testing
4. Behavior hasn't changed
5. Performance is similar or better
